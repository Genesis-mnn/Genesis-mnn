# 贡献指南

感谢你对 Genesis（创世纪）项目的关注！我们欢迎任何形式的贡献，包括代码、文档、问题反馈和创意讨论。

## 贡献流程

1. **Fork 与 clone**：将仓库 fork 到你的 GitHub 账户，并 clone 到本地。
2. **创建分支**：基于 `main` 分支创建特性/修复分支，命名如 `feat/your-feature` 或 `fix/issue-123`。
3. **编写代码**：遵循项目的代码规范和术语中立要求，添加必要的测试和文档。
4. **提交 PR**：推送分支，并在 GitHub 上发起 Pull Request。PR 描述中请说明改动动机、实现方式和可能的副作用。
5. **代码审查**：核心维护者将进行审查，可能需要调整。请保持友善沟通。

你也可以通过 [Issues](https://github.com/Genesis-mnn/Genesis-mnn/issues) 直接报告 bug、提出功能建议或参与设计讨论。

## 代码规范

- **PEP 8**：请遵循 Python 官方代码风格，推荐使用 `black` 或 `flake8` 进行格式化/检查。
- **类型注解**：所有公共 API 必须包含完整的类型注解（type hints），函数和方法的 docstring 应清晰说明参数和返回值。
- **术语中立**：严格遵守项目的《术语中立宪章》。所有内部变量、方法名、注释和文档**禁止使用人类生物化学或情感相关词汇**（如“情绪”、“奖赏激素”等），必须采用中性术语（如“状态”、“调制原”、“响应子”）。现有常量如 `MODULATOR_REWARD = "奖赏信号"` 应沿用。
- **测试**：核心功能（神经元、突触、胶质细胞、调制场）应提供单元测试或集成测试。运行现有测试套件确保未引入回退。
- **文档字符串**：模块、类和方法使用多行文档字符串（遵循 Google 或 NumPy 风格）。

## PR 审查要求

你的 Pull Request 需要通过以下检查：

1. **所有测试通过**（`pytest` 或无报错）。
2. **未破坏核心公理的一致性**：提交的代码不得引入全局反向传播、外部预定义损失函数或破坏“内在性优先”、“共鸣取代权重”等基本承诺。
3. **包含必要的单元测试**（对新增模块、关键逻辑）。
4. **公共 API 文档串完整**，类型注解正确。
5. **术语中立**：审查者将专门检查是否误用违禁词汇。

若 PR 包含对框架核心行为（如神经元动力学、突触可塑性规则）的修改，需附上简短的实验记录或理论对比，以说明其对数字生命行为的预期影响。

我们鼓励所有贡献者先通过 Issue 讨论重大变更，以减少无用功。欢迎每一位建设者的加入！
